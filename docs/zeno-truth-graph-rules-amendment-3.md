# Zeno Truth Graph — 呈现规则修订案 №3(带内多列 · 边路由三分法)

> 读者:Lixian(提取层)、Codex(前端实现)。
> 状态:修订案,基于 v1 基线(`zeno-truth-graph-rules.md`)与修订案 №1、№2。并入 v2 前,与前述文件冲突处以本文件为准。

---

## 0. 本修订案不改变什么

- **带 = 生命周期**(修订案 №1 §2)完全不变:锚 / 前提 / 已定 / 待决 / 已排除,纵向顺序即叙事顺序。
- **待决带仍是唯一视觉焦点**,仍用完整卡片(§4 密度分档不变)。
- **安静边的范围**(修订案 №2)不变:仅 `depends_on` / `implies`、仅同 topic、仅两端顶层且可见。
- 绿红唯一语义、四形状、色盲冗余信号、truth 只读(v1 §4、§7)全部保留。

## 1. 诊断(为什么要修订)

修订案 №1 把 topic 内布局改成五条横向带,每带内部仍是**每行一个节点的单列**。这在节点少时成立,但案例一复杂就暴露两个问题:

1. **纵向无限增长**。已定带是增长最快的带,单列意味着项目越老、可见区越往下滚,「恒定可见区」的承诺(修订案 №1 §4)靠折叠已排除项撑不住。
2. **线条堆叠**。多个前提被同一个决定依赖时,单列让它们纵向拉开很远,边全部挤进左 gutter 的平行通道,跨距越长越难读——恰恰是修订案 №2 想避免的观感。

**根因:单列强制「一个节点占满一行」,浪费了横向空间,又把本该并排的并列关系拉成了纵向长距。**

## 2. 带内多列:前提带与已定带

**前提带(premises)与已定带(settled)改为横向 flex-wrap 多列**,其余三带(锚 / 待决 / 已排除)保持整宽。

| 带 | 形态 | 为什么 |
|---|---|---|
| 锚 | 整宽 | 一个 topic 通常一个锚,它是标题级元素 |
| **前提** | **多列 cell** | 前提是依赖边指向的目标;并排后「一个决定依赖两个前提」的①②汇聚自然成形,而非纵向拉长 |
| 已定 | **多列 cell** | 增长最快的带;多列让可见区近似恒定 |
| 待决 | 整宽卡片 | 卡片是「请注意我」的形态(修订案 №1 §4),缩窄会削弱它 |
| 已排除 | 整宽(折叠内) | 抽屉内容,不值得版面优化 |

**关键约束:多列只是密度优化,不承载任何新语义。** 带内顺序仍是创建序;阅读序为**先上后下、同行先左后右**。列数由容器宽度决定(`--z-cell-basis` 220px 自然 2–3 列),不由内容重要性决定——重要性已经由「在哪条带」表达完了。

cell 用 `flex-basis` + `grow` 做等宽列(而非内容自适应宽),这样:
- 换行只取决于容器宽度,行为可预测;
- 同一 flex 行内的 cell 因默认 `align-items: stretch` **等高**——这是 §3 视觉行聚类算法成立的前提,**不要覆写为 `items-start`**。

窄容器下 `min-width` 使其自然退化为单列,无需断点。

## 3. 边路由三分法

多列后,「一律从左缘出、走左 gutter」不再适用。按**两端的相对位置**分三种路由:

| 路由 | 判据 | 形态 |
|---|---|---|
| **tight** | 同一视觉行、水平相邻(间隙 ≤ `--z-tight-arrow-gap` 44px)、且两者之间无第三个 cell | 右缘→左缘的**水平直线**短箭头(v1 §4.4:同行短箭头保持直线,不拐弯) |
| **step** | 父行与子行相邻,且**至少一端所在行含 ≥2 个 cell**,且两端水平中心偏移 ≤ `stepMaxJog` | 父底边出 → 行间走廊水平 lane 平移 → 子顶边入(箭头朝下) |
| **gutter** | 其余全部(长跨、上行边、同行不相邻、两端均为整宽行) | 修订案 №2 原样:左 gutter 内独立垂直通道 |

### 3.1 为什么 step 要求「至少一端多列」

两个**整宽行**上下堆叠时,它们之间只有 2–8px 间隙,走 step 会画出一根挤在缝里的短竖线,反而不如绕左 gutter 清楚。修订案 №2 的 gutter 路由是已验证的行为,**单列布局下必须原样保留**。step 只在多列真实存在时才介入。

### 3.2 降级规则

- step 的两端水平偏移过大(> `stepMaxJog`,约 1.5×cell 宽)→ 横向 jog 太长会显乱 → **降级 gutter**。
- 同行但间隙过大或中间夹着第三个 cell → **降级 gutter**(直线会横穿无关节点)。

### 3.3 双轨通道分配

两套通道各自独立打包,规则同源(**短跨占内侧**,区间不重叠即可复用同一轨道):

- **gutter 垂直通道**:按边的纵向区间打包;轨道 x 从 gutter 右缘向左推进(`--z-lane-channel-gap` 间距)。
- **行间水平走廊 lane**:按边的横向区间在**所属走廊内**打包;lane 紧贴子行上缘,向走廊内侧向上堆叠(`--z-lane-hlane-gap` 间距)。走廊容量不足时 clamp 到最外 lane,接受轻微重叠——与 gutter 的 `maxChannels` 同一处理哲学:**宁可轻微重叠,不可画到别的行上去**。

平局一律按 `edgeId` 字典序决,保证同一份数据每次渲染结果一致(确定性布局,v1 §1.1)。

## 4. ①② 汇聚编号的排序更新

v1 §5.3 规定多个并行前提在入口标①②。单列时「按父节点 top 排序」等价于阅读序;多列后必须改为:

> **按 `(视觉行号, 左缘 x)` 字典序** —— 先上后下,同行先左后右。

即用户从左到右扫过一行前提时,序号就是①②③。step 路由的入边沿子节点**顶边水平分散**(不再是纵向分散),徽章落在箭头正上方。

## 5. 新增 token

全部在 `docs/zeno-truth-graph-tokens.css`,组件禁止硬编码:

| token | 默认值 | 用途 |
|---|---|---|
| `--z-cell-basis` | 220px | 多列 cell 的 flex-basis(自然 2–3 列) |
| `--z-cell-min-w` | 170px | cell 最小宽(窄容器退化为单列) |
| `--z-cell-max-w` | 360px | cell 最大宽(单列时不铺满到难读) |
| `--z-cell-gap-x` | 8px | 同行 cell 列间距 |
| `--z-cell-gap-y` | 8px | 行间距 = 行间走廊高度下限 |
| `--z-lane-hlane-gap` | 6px | 走廊内相邻水平 lane 间距 |

几何模块里的数值常量是这些 token 的镜像(SVG path 数学读不到 CSS 变量),**tokens 文件是唯一真相来源**;改 token 时必须同步 `lane-edges.tsx` 的 `GEOMETRY`。

## 6. 测量层的一个必要加固

多列下,一个 cell 可能**在容器外形不变的情况下换列或改宽**。因此 `measureRows` 的 signature 必须包含 `width`——只有 top/left/height 时,「等高换列」会被误判为无变化而跳过重算。

## 附:本修订案新增决策清单

1. 前提带与已定带改为 flex-wrap 多列;锚/待决/已排除保持整宽
2. 多列仅为密度优化,不承载语义;带内序仍为创建序,阅读序=先上后下先左后右
3. cell 用等宽 flex-basis(非内容自适应),同行等高是几何前提
4. 边路由三分法:tight(同行相邻直线)/ step(相邻行经走廊)/ gutter(其余,修订案 №2 原样)
5. step 要求至少一端所在行为多列——单列布局完整保留修订案 №2 行为
6. 偏移过大 / 中间夹节点 → 降级 gutter
7. 双轨通道:gutter 垂直 + 走廊水平,同为「短跨内侧、区间不重叠可复用」,平局按 edgeId 决
8. ①② 排序改为 (视觉行号, 左缘 x) 字典序
9. 测量 signature 必须含 width
