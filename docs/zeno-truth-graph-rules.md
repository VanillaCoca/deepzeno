# Zeno Truth Graph — 呈现规则与设计制度 (v1)

> 读者:Lixian(提取层 / 图表决策逻辑)、Codex(前端实现)。
> 状态:V1 设计基线。本文件是判断规则 + 实现约束,不是会议纪要。每条规则尽量写清「是什么 / 为什么 / 怎么执行」。

---

## 0. 最高信条(凌驾一切)

**大部分人看一眼就知道什么意思 —— 易学易用,是最高优先级的设计信条。**

Zeno 的用户大量不是程序员(移民规划者、做课题的研究者、产品经理、独立开发者)。任何"需要先学一套概念/符号才能看懂"的设计,无论多精确、多专业,一律不采用。

这条信条用来裁决本文件中所有后续规则的冲突:当"灵活/精确/美观"与"看一眼就懂"冲突时,永远选后者。

附属原则(继承自产品宪法):
- **宁朴素勿炫技** —— 元素只在「改变理解」时才出现;不改变理解的元素一律砍掉。(与铁律「宁漏勿错」同构)
- **统一优先于灵活** —— 见 §3,变的是图的"种类",不变的是符号的"含义"。
- **调查全自动,真相确认制,确认是稀缺资源** —— Zeno 主动产出的任何东西(开题播种、调研简报、巡检告警)也必须一眼看懂,否则砍掉;确认请求永远稀缺,确认必须是思考行为。(继承自宪法修订案 №1,Iron Law 0 与 §2)

## 1. 核心架构:两层视图

Truth Graph 不是「一张万能图」,而是两层,各司其职:

| 层 | 职责 | 何时出现 |
|---|---|---|
| **全貌层 (Overview)** | 看「项目沉淀了什么、分几个主题、各自什么状态」 | 默认态,始终在场 |
| **单链层 (Chain)** | 看「某一个判断是怎么一步步被推导出来的」 | 用户选中某节点后 |

**关键设计:这两层不是「跳转」关系,是「同屏并置」关系。** 见 §2。

### 1.1 全貌层规则

- 按 **Topic 分组**(Topic 是 schema 一等对象)。每个 Topic 是一个容器,内含若干节点。
- **默认不画任何关系线。** 这是 V1 的明确决策,不是偷懒。
  - 为什么:用户打开时唯一需要的信息是「有哪些判断、分几类、什么状态」;关系是「想深入某一个时」才需要的。选择前画全部关系线 = 视觉灾难(无法有效展示)+ 违反最高信条。这是渐进式披露的最高纪律:默认态零关系负担。
  - 例外:见 §2,选中某节点后,**只点亮该链相关的线**,从不点亮全部。
- 全貌层显示:节点标题(短)、节点状态(靠颜色/形状,见 §4)、Topic 分组。
- 节点位置 **稳定**(确定性布局)。同一项目每次打开,全貌形状一致 —— 这服务于「判断状态的连续性」这一产品本质,用户的空间记忆不能每次被打乱。
- 新 truth 确认进入 SSOT 时,全貌**增量重排**:布局引擎重算,但已有节点尽量保持原位(stable incremental layout)。

### 1.2 单链层规则

- 用户选中某节点 → 抽出它的**完整上游推导链**(该节点 + 所有祖先),竖直从上到下铺成流程图。
- 「最短路径查看关联」的正解 = **把这条链从网里拽出来、拉直**,而不是把全貌的线画得更聪明。
- 端点锚定 + 条件编号,见 §5。
- 汇聚处理(一个节点依赖多条独立上游链),见 §5.3。

## 2. 交互模式:同屏并置(桌面),不跳转

**结论:点进新页面再返回(drill-down + back)是反模式,V1 不用。** 此结论有市场验证支撑:专业图可视化领域一致采用「原地聚焦 + 无关项推到背景」而非「换页」,核心原则是「维持上下文(maintain context)」。

### 2.1 桌面端(V1 唯一目标平台)

**布局:左侧全貌(固定)+ 右侧单链(随选刷新)。**

- 点全貌任意节点 → 右侧把它的推导链铺成竖直流程图;同时左侧**只点亮这条链**(相关节点+线变高亮,其余淡化)。
- **左侧全貌图始终不动**(空间记忆零破坏)。用户可直接点下一个节点,右侧跟着换,左侧高亮跟着切 —— 全程不离开,支持「连续追查」而非「进-看-退-再进」的离散跳转。
- 左图(位置)与右图(推导顺序)互为印证:左边告诉你「它在项目里的位置」,右边告诉你「它怎么来的」。

为什么这是对的:判断状态的探索本质是连续的(看完一条链自然想看相邻的)。换页会把连续探索切成离散动作,长期使用必然累。

### 2.2 移动端

**V1 不做,V2 不做,目前完全无计划。**

理由:Zeno 的目标用户(独立开发者、做严肃项目者)维护判断状态几乎一定在电脑上做。移动端不投入工程。本文件所有交互规则默认桌面/宽屏。

## 3. 图表类型:形态自适应,语言统一

### 3.1 两个维度必须分开对待

Craft Mermaid 提供的「不同」其实是两个维度,Zeno 对它们态度相反:

| 维度 | 是什么 | Zeno 的态度 |
|---|---|---|
| **图表类型** (flowchart / state / sequence …) | 由「内容本质形态」决定 | **自动判断、自动切换** |
| **配色 / 主题** (Tokyo Night / Nord / Dracula …) | 审美风格 | **永不自动变 —— 全产品统一一套** |

**一句话:Zeno 会自动画出不同种类的图,但所有图共用同一套视觉语言(同样的颜色、同样的符号含义)。变的是图的种类,不变的是符号的含义。**

比喻:同一个人,有时说中文有时说英文(图表类型按内容切),但人格、语气始终如一(视觉语言统一)。

为什么配色不能变:若每个项目配色风格不同,用户每开新项目都要重新适应「这个主题里红是什么意思」—— 这正是「要先学一堆概念」,违反最高信条。统一才是易学。这也是「判断状态连续性」在视觉层的体现。

唯一允许的变体:**深色/浅色模式**(跟随系统设置)。它不改变任何语义(绿还是绿、红还是红,只是底色变),纯护眼。用 Craft 式双色推导(给定 bg + fg 两色,整套配色自动适配明暗)实现。

### 3.2 图表类型选择决策树(给 Lixian 的判断规则)

Zeno 先判断「这份 truth(或这个 Topic 内的链)是什么形态」,再选图:

| 内容本质 | 用 | 触发信号 |
|---|---|---|
| 一连串「做了A才能做B」的依赖 | **流程图(默认)** | 节点间是 requires / leads_to,有明确先后 |
| 有岔路/条件(「如果…就…」) | **流程图 + 菱形 + 绿红分支** | 出现条件分叉,一条接受一条拒绝 |
| 一个东西在几种状态间转换 | **状态流转图**(去掉 `[*]` 等专业符号,起点直接写文字) | 「草稿→评审中→已批准」这类生命周期 |
| 按时间顺序的多方互动(谁先谁后) | **时序图** | 多个参与方 + 时间序列消息 |
| 纯粹的量的对比 | **柱状/折线图** | 数值比较 |

### 3.3 同一项目内允许多种图表类型

**允许,且支持。** 一个项目里不同 Topic 形态不同(「技术栈」是依赖链 → 流程图;某个走审批的 Topic 是状态流转 → 状态图)。点进不同 Topic 的链,右侧可能呈现不同类型的图。

- 代价:用户在同一项目看到两种图,轻微违反「统一」。**接受这个代价** —— 一切以易懂为最高,灵活让位给「这个内容用哪种图最好懂」。
- 护栏:**默认收敛到流程链**,只有当某 Topic 内容明显是状态流转(或其他形态)时才切换。「允许但保守」。
- **再次强调护栏的边界:灵活的对象是图表"类型(骨架形态)",不是"视觉语言"。** 无论哪种图,绿永远是已确认、红永远是已否决、▷永远是起点、虚线永远是未定。用户学一次配色和符号,走遍所有图。这是「易学」的真正实现:不是「每张图都一样」,而是「所有图说同一种语言」。

### 3.4 永久淘汰的元素(违反最高信条)

任何「符号本身需要专门学习才知道含义」的元素,一律不用:

- **Class 图全部 UML 关系符号**(`<|--` 继承、`*--` 组合、`o--` 聚合等)—— 连程序员都要查。淘汰。
- **ER 图鱼尾符号**(`||--o{` cardinality)—— 数据库专业记号。淘汰。
- **State 图 `[*]` 伪状态** —— 需学习的约定。淘汰该符号(状态机概念保留,起点用文字「开始/草稿」)。
- 通则:鱼尾、空心三角、实心菱形(UML 含义)、`[*]` 等,全部违反「看一眼就懂」。

## 4. 统一视觉语言(铁律,跨所有图不变)

### 4.1 四种节点形状(映射角色,只用最直觉的四种)

| 形状 | 含义 | 直觉来源 |
|---|---|---|
| 胶囊/圆角矩形 `([ ])` | **起点 / 入口** | 柔和,像「开始」按钮 |
| 普通矩形 `[ ]` | **一个决定 / 一件事**(默认) | 默认形态 |
| 菱形 `{ }` | **一个待决问题 / 岔路口** | 菱形=要做选择,路标级通用直觉 |
| 圆形 `(( ))` | **终点 / 最终结果** | 终态 |

超过这四种(数据库圆柱、六边形 subroutine 等)开始需要学习,**一律不用**。

### 4.2 颜色(只在有语义差异时出现,其余靠透明度/灰阶)

绿/红/灰是主力,语义**唯一且固定**:

| 颜色 | 唯一含义 | 备注 |
|---|---|---|
| 绿 | **已确认 / 已采纳** | 不表示「推荐」 |
| 红 | **已否决 / 已推翻 / 有冲突** | 不表示「避免建议」 |
| 中性灰 | 普通依赖 / 进行中 | 节点默认 |
| 琥珀 | **待决问题(open question)** + **前提状态异常**(见 §6) | 「需要你关注/处理」 |

- **绿红收窄成唯一语义:已确认 vs 已否决。** 「推荐 vs 避免」这种建议性语义**不用颜色**,用文字标注。这样红绿在所有图里永远同一个意思,杜绝歧义。
- 借红绿灯的全人类直觉,非技术用户瞬间懂 —— 这是最高信条的最强武器。
- fact / decision 默认**不上自己的颜色**,只是前景色的不同深浅(节点近透明,靠描边+文字定义,Craft 式 3% 填充)。颜色只在状态有语义差异时出现。

### 4.3 线型(实/虚/粗,物理直觉,不需学)

| 线型 | 含义 |
|---|---|
| 实线 | 确定依赖 |
| 虚线 | 弱关系 / 回退重试 / **前提未定** |
| 粗线 | 关键路径(慎用) |

### 4.4 连线路由:正交通道(直角拐弯),不用弧线

- 跨行/跨节点的长依赖线用**正交路由**:出节点走横 → 圆角直角 → 沿独立垂直轨道走竖 → 圆角直角 → 进目标。
- **每条线分配独立的垂直 channel**,平行排列、永不重叠 —— 这是直角胜过弧线的实质原因(弧线多了无法平行堆叠、会缠绕)。
- 轨道按跨度排序:跨距短的走内侧,长的走外侧(减少交叉的标准启发式)。
- 拐角用小圆角(非尖直角),保留清晰但不生硬。
- **tight 关系的横向短箭头(同行内)保持直线**,太短不需拐弯。直角路由只用于跨行长线。
- 这套与 Zeno 整体语言(CLI 树、hairline 边框、直角节点、Claude Code 极简)一致;弧线偏「流动/关系网络」(Obsidian 调性),是 Zeno 刻意拒绝的。

### 4.5 Craft 美学的精确数值(给 Codex)

源自 Craft beautiful-mermaid 的双色推导。给定 bg + fg 两色,其余按比例 color-mix 推导:

- 节点填充:fg 混 **3%**(近透明,像纸上的字,不像按钮)
- 节点描边:fg 混 **20%**(decision);fact 更淡
- 连接线:fg 混 **~22%**(线退到背景)
- 箭头头:fg 混 **~52-55%**(箭头是前景)
- 边标签:fg 混 **~45%**
- 次级文字:fg 混 60%

设计哲学(Apple / Craft 式极简,直接迁移):删掉所有不承载信息的线条 —— 无轴线、无网格、Topic 之间靠**留白**而非边框分隔(全貌的 Topic 容器可保留极淡描边,但不做成「仪表盘卡片」)。

### 4.6 聚焦:三级透明度 + 平滑过渡,不重绘

选中某节点时,用 CSS opacity 过渡(~0.28s),**不重新渲染整张图**:

- 聚焦节点:opacity 1
- 直接相关(一跳):opacity ~0.55
- 无关:opacity ~0.14(淡到极弱但**不消失**,仍在原位,保住空间记忆)

这是「用透明度分三层注意力」—— Craft 静态美学之外、Zeno 需额外设计的动态聚焦语言。丝滑渐变 vs 闪一下,是「高级」与「卡顿」的分界。

### 4.7 可访问性:红绿色盲

红绿色盲约占男性 8%,而开发者用户男性比例高。**纯靠红绿区分「确认/否决」对他们失效。** V1 必须在红绿之外叠加非颜色信号:

- 已否决/已推翻:加**删除线**(line-through)
- 已确认:可加 **✓** 标记或加粗描边
- 即任何依赖绿/红的语义,都要有一个不依赖颜色的冗余信号(形状/符号/线型)。

## 5. 步骤顺序:端点锚定,条件编号

「点击节点后不确定哪里是开始」的根因不是缺序号,而是「从上往下读」的约定不够强。解药是**端点锚定**,不是全程编号。

### 5.1 默认不给每一步编号

- 靠「起点/终点的形状 + 措辞」建立方向感。
- 给纯线性链每步编号 = 冗余(箭头已说明顺序)= 违反「不堆元素」。

### 5.2 永远显式标「起点」

- 起点(最初的 fact/约束)= 胶囊形 + `▷` + 上方小字「从这里开始」。这是方向感最便宜、最有效的来源。
- 终点(用户选中的节点)= 圆角加粗 + 上方「你选中的」标记。
- 中间节点保持朴素。

### 5.3 序号只在三种情况出现

1. **汇聚点的并行前提**:一个节点同时依赖多条独立上游链时,在它入口标 ①②,告诉用户「这几个是并行前提,都要完成」。序号在此承载了「箭头表达不了的信息」(并行 vs 串行)。
2. **超长链**(如 >6 步,容易看丢位置):给关键节点编号。
3. **手册型 truth**(内容本质就是严格按序执行的操作步骤,如换汇实际办理流程):序号是内容的一部分,该加。

### 5.4 汇聚的视觉表达

单链层遇到汇聚点(目标节点依赖多条独立上游链)时:
- 多条链各自竖直排布,在目标节点处汇入。
- 每条入边标 ①②…,目标上方/详情注明「需先完成 ①②」。
- 每条链的线型/颜色独立编码其状态(如一条绿实线=已确定的前置链,一条琥珀虚线=含未定前提的链)。用户一眼看出「哪条腿稳了、哪条还悬着」。

## 6. Facts & Assumptions(前提)的处理

### 6.1 核心区分:前提 ≠ 判断

用户(尤其非技术用户)心智里,「前提」和「决定」是两种东西:
- **前提**(fact / assumption)= 「我们假定…」—— 可能被推翻的地基。
- **决定**(decision)= 「我们定了…」—— 已采纳的判断。

**把两者画成相同的方框,会让用户分不清「哪些是我已定的、哪些只是我假设的」—— 这是判断状态里最危险的混淆。** 例:移民规划者必须一眼分清「走美元路径」(我定的决策)vs「假设人民币管制政策不变」(前提)—— 后者一旦变,整条链要重审。

这个区分承载 Zeno 核心价值:让用户清楚知道判断**建立在哪些假设之上**,以及某假设动摇时哪些判断受影响。

### 6.2 作用域:Global vs Local

- **Global**:作用于整个项目,或整个 Topic 的成立前提。
- **Local**:仅作用于某一个/某几个节点的前提。

成熟领域验证的统一原则(数学公理栏 / 合同定义章 / 代码作用域 / Toulmin warrant):**前提的作用域决定它的视觉位置 —— 全局前提「声明一次、默认隐含、按需展开」;局部前提「就近显示在它管辖的节点旁」。**

### 6.3 落地规则

| 类型 | 处理 |
|---|---|
| **Global fact/assumption** | **不进 graph 当节点。** 放全貌顶部 / Topic 标题下的**折叠条**(默认折叠成一行,如「基于 3 条前提」,点开看全文)。对应数学公理栏、合同定义章 —— 始终在场、随时可查,但不占判断的视觉主体。 |
| **Local(单节点)** | **不单独成节点**,作为该 decision 详情里的「前提说明」。 |
| **Local(跨多节点)** | 才有资格在图里露出,但**不当普通节点** —— 用视觉上明显不同的「前提标记」(小的、虚线的标签,挂在受影响节点上方),区别于实心 decision 节点。 |

- 任何前提**不进绿红体系**。绿红专属判断状态(已确认/已否决)。前提有自己的状态:**「成立中 / 已失效」**,用**琥珀 / 图标**单独表达。
  - 为什么:若前提失效也用红,会与「判断被否决」混淆。琥珀(与 open question 同族,都是「需关注/处理」)或图标(⚠ / 虚线变实线)单独承载。

### 6.4 假设失效预警(核心机制)

这是 facts/assumptions 区分带来的杀手级功能种子。

**触发(唯一入口):用户手动把某个 assumption 标记为「已失效」。**
- **允许且必须允许**用户手动标记。提供状态切换按钮(「成立中 ⇄ 已失效」)。
- 为什么是用户手动而非 Zeno 自动:Zeno 不读新闻、不知道用户现实世界发生了什么。**只有用户能报告「外部世界变了」。** 没有这个按钮,「假设失效预警」根本无从触发。
- **关键边界:此按钮只切换 assumption 自身状态 + 触发下游高亮,绝不直接修改任何 decision。** 它是「报告现实变化」(火警按钮),不是「修改判断」(重新设计房子)。报告 ≠ 修改,所以不违反「truth 只读」铁律。

**响应:只高亮提示,不自动降级。**
- assumption 标记失效 → 在受影响的下游 decision 上加视觉提示(淡的「前提已变」标记 + 顶部一行「N 个判断建立在已失效的前提上」)。
- **不自动改任何 decision 状态**(不自动打回 candidate)。
- 为什么:**假设失效 ≠ 判断失效。** 一个判断可能有多个支撑,某前提倒了未必垮 —— 只有用户自己知道。Zeno 职责是「提醒有风险」,不是「替你判定有罪」。对应铁律「宁漏勿错,trust-preserving > recall maximization」:宁可只提示(可能其实没事),也不自动降级(可能误杀仍成立的判断)。

## 7. Truth 只读,所有判断变更回 Sandbox(铁律加固)

### 7.1 世界观(设计哲学,写在最前)

Zeno 假设:从现在起,无论 PhD 科研、产品设计、公司运营/团队协作,都在 LLM 深度参与下进行。用户是在与多个大模型协作探讨的基础上产生 SSOT。**SSOT 由 Zeno 产生,也应由 Zeno 维护;用户通过与 Zeno 探讨来修改(或直接下达修改命令),不手动调整。**

用户的 mindset 也不倾向手动调整:发现某判断与设想不符时,他们应做的是 **bring back to sandbox 重新探讨**,而非手动修改 truth。

### 7.2 V1 不给手动调整图的能力

- 不给「手动标两节点相关」「手动改某条线关系类型」等。
- 布局和关系**全由 AI 判断**,用户只读。
- 理由:见 §7.1。先看 AI 判得够不够准。
- 注:这与 §6.4 的「标记 assumption 失效」不冲突 —— 那是报告外部现实,不是手动编辑判断/布局。

### 7.3 不做「快速忽视」—— 结构上就错,不是 V1 缓做

**Truth graph 上不提供任何修改 decision 状态的操作,包括「快速忽视前提变化影响 + 写 memo」。**

为什么是「永远不做」而非「V1 先不做」:
- 「快速忽视 + 写 memo 说明理由」的本质 = **一次微型判断讨论**(「这前提变了,但我认为判断仍成立,因为……」)。这个「因为……」正是 re-evaluation,正是 sandbox 该做的事。
- 它是 **sandbox 的残缺阉割版** —— 在错误的地方(truth graph)做了本该在 sandbox 做的事(重新评估判断)。
- 后果:判断的 re-evaluation 散落两处(sandbox 完整讨论 + truth 上随手 memo),SSOT 立刻不「single」—— 依据一部分在 sandbox 历史,一部分在散落忽视备注里。直接违反「SSOT 必须由探讨产生、由探讨维护,只有一个产生它的地方」。
- 它在 truth 层开了个做判断的后门,会越用越大(用户图省事,能忽视就不去 sandbox,SSOT 退化成一堆带忽视备注的旧判断)。

### 7.4 正确流程

用户看到「前提已变」高亮后,两个动作**都不在 truth 上直接做**:
1. 觉得需要重审 → bring back to sandbox 讨论,结论(维持或改)带完整理由回 truth。
2. 觉得不影响 → 这也是个判断,也该有依据 → 同样去 sandbox 快速确认,Zeno 记录「用户确认 X 前提变化不影响此判断,理由:……」,然后消除高亮。

第 2 种情况和「快速忽视带 memo」想做的事完全一样,但**发生在 sandbox 里,理由进入正规判断历史**,而非散落 truth 上。同样的用户意图,放对地方=维护 SSOT 完整,放错地方=后门。

### 7.5 产品该投入的

不是「忽视」功能,而是**「从高亮提示一键 bring back to sandbox」的顺滑入口** —— 顺滑到用户不觉得「去 sandbox 讨论」是负担。负担感的解药是优化 sandbox 入口体验,不是在 truth 开后门。

## 8. 杂项规则(已定)

- **节点标题超长(>60 字)**:截断到一行 + 详情看全文。不换行,不中间省略号。
- **空项目 / 仅 1-2 节点**:不画图,直接列表。图在节点数 ≥ 阈值才出现。
- **关系标签语言**:跟随用户界面语言(中文显示「需要」,英文显示 needs),不混用。
- **supersede(被取代)节点**:全貌默认**藏**;单链里若某节点取代了旧的,在**详情**显示「取代了 X」。越简洁越好。

## 9. 技术底座(渲染 / 导出,非交互层)

- **交互层(全貌聚焦、单链抽取、三级透明度)自研** —— 这是 Zeno 差异化,Mermaid 做不了。
- **渲染/导出底座用 Craft beautiful-mermaid**(MIT,1.0+,内置 ELK.js 正交布局):
  - 项目全图概览 / 分享导出 → `renderMermaidSVG`(美观)
  - MCP read 给下游 agent → `renderMermaidASCII`(agent 易读,事实标准,零解析成本)
  - agent 提交的图解析回结构 → `parseMermaid`(供 submit_candidate 链路)
- **数据流:`ir_nodes` → 生成 Mermaid 文本(Zeno 掌握的中间层)→ 三个出口:SVG(给人)/ ASCII(给 agent)/ 自研交互层(给探索)。** Mermaid 文本是 truth 的「可移植表达」,这一层自有,渲染外包。
- 待验证工程点:用真实 schema 跑 20 节点 + 5 子图样例,验证 Mermaid `subgraph` + 子图内方向覆盖的布局质量,再决定全图视图是否完全依赖它。
